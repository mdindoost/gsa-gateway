## 1. Problem & the reframing that drives this design
Routing is spread across four hand-maintained mechanisms — a keyword cue pre-gate
(`message_handler._try_structured`), a regex router (`router.route`), a regex intent detector, and a RAG
threshold. They mis-route natural phrasings across **all** route families.

**Key insight (Claude-web): "deterministic" ≠ "exact."**
- **Exactness** is a property of the **answer path** — guaranteed by "the number/roster comes from
  parameterized SQL; the LLM never computes." Non-negotiable.
- **Determinism** is a property of the **routing decision**, and that determinism IS the brittleness (a
  regex router is a lookup table over phrasings; every phrasing it didn't enumerate is a miss).
Since exactness is protected downstream at the SQL boundary, the router is freed to use the most
**accurate** classifier we can run where natural-language generalization is needed — **but only where it
genuinely beats lexical rules** (see §5: that is the coarse family boundary + phrasing robustness, NOT
within-KG skill selection, which is lexical and stays deterministic).

## 2. Goals (TWO CO-EQUAL)
- **G1 — Route the whole question space well.** Best route for ANY grad-student question across every
  family (structured KG, RAG, live njit.edu, conversational/commands, clarify), robust to phrasing.
- **G2 — Never let an exact/structured query get a fuzzy answer.** Counts/rankings/rosters/metrics/roles
  come ONLY from deterministic parameterized SQL. The LLM may classify/parse/fill slots, never compute
  or invent numbers, names, or SQL. (Anti-fabrication / honest-partial.)

## 3. Core principle — separate FUZZY routing from EXACT answering
Re-bucket by **property**, not order: everything probabilistic (coarse intent classification, Phase-2
slot proposal) on one side; everything exact (entity/metric resolution, lexical skill selection within
KG, SQL execution, numeric compose-suppression) on the other. **The line IS the exactness guarantee.** A
misclassification can only route to clarify, RAG, or a different *validated* skill — never a wrong number.

## 4. Route taxonomy (the whole picture — corrected to what the schema can route)
- **Structured KG (exact, ~18 skills)** — entity_card, research_of_person, metric_of_person,
  link_of_person, faculty_in_department, people_in_org, officers_in_org, people_by_role,
  people_by_research_area, count_people_by_research_area, areas_in_org, area_counts,
  faculty_areas_in_department, people_by_area_tag, top_people_by_metric, org_departments, people_by_name,
  person_disambig.
- **Semantic RAG — only 3 ROUTABLE behavioral outcomes today** (RAG-expert C-1 fix): `food-handler`
  (→ `get_food_events`), `event-filtered` (`source_type='event_info'`), and `general-RAG`. The richer
  "content families" (funding, constitution, MMI, club rules, programs, resources, policies, FAQ) are
  **content, not routes** — the schema has no `type` to filter them on; they all live in `general-RAG`
  and are separated by *retrieval quality*, not routing. **Fine-grained per-corpus RAG routing is
  DEFERRED** (§14). NOTE: setting any `source_type` filter **disables BM25** (vector-only) in the
  retriever → measure recall impact (§9); do not filter sparse-term queries (acronyms/award names).
- **Live njit.edu fallback** — long-tail facts, extractive/grounded.
- **Commands & conversational** — clear/mode-switch/greeting/identity/thanks/farewell (deterministic
  whole-message commands); food/social are content → the RAG outcomes above.
- **Clarify** — FIRST-CLASS route for underspecified queries; deterministic template, no LLM.

## 5. Architecture — one unified Router: classify-then-resolve
```
message
 0. COMMAND layer (deterministic, whole-message, high-precision regex): clear / mode / greeting /
    identity / thanks / farewell → handle directly. (Precedence preserved from intent_detector; §9 tests.)
 1. FAST-PATH (deterministic, FIRST-CLASS — sized from the Phase-0 head distribution to cover COMMON KG
    intents, not just a stereotyped handful): exact lexical match → run SQL skill with ZERO encode.
 2. COARSE CLASSIFIER (fuzzy, learned-untrained = embedding exemplar-NN; reuse nomic + sqlite-vec):
    message → { family ∈ [KG, RAG-outcome, live, clarify], score, margin }. ONE encode per message
    (§8 invariant). Decides the FAMILY, not the KG skill.
 3. RESOLVER / VALIDATOR (deterministic, EXACT — the line):
    - family=KG → run route()-style DETERMINISTIC lexical skill-selection + its NEGATIVE GUARDS
      (_OFFICER_PROCESS / _LEADERSHIP_PROCESS / _has_child_departments / _is_bare_name / ≤4-tok surname)
      as dispatch preconditions → resolve+validate slots (candidate-IDs only) → parameterized SQL.
      Empty/invalid → RAG (NEVER a fabricated honest-partial). Ambiguous entity → clarify.
      TERMINAL-SKILL GUARD: link_of_person / metric_of_person emit honest-partial on empty BY DESIGN, so
      they require the regex link/metric-word precondition (kept) before dispatch — a learned family
      signal alone must NOT trigger them (else confident false honest-partial). [RAG-expert SF-1]
    - family=RAG-outcome → set the (food/event/general) behavior → RAG.
    - family=live → sanitized public query → njit.edu extractive fallback.
    - family=clarify → deterministic template naming the missing slot (no LLM).
 4. INVERSE-FN GUARD [router Con-5]: if the classifier picked a RAG family BUT the query also trips a
    high-precision KG cue (resolved org/person AND a metric/role/faculty cue), run the deterministic
    resolver anyway and PREFER its validated result. Restores the safety detector-first had for free.
 5. LLM SLOT-RECOVERY — Phase 2, sub-flag, OFF at first ship. Only when family=KG but the deterministic
    resolver can't extract slots: one Ollama format=json call, candidate-IDs-only, DB-validated,
    confidence floor, ambiguity→clarify, else→RAG. Additive; zero-regression when off.
```
**Default Phase-1 mechanism = coarse-family classifier + deterministic within-KG skill selection.** Both
experts' prior: exemplar-NN cannot reliably separate adjacent KG skills (faculty/people/officers/role
differ by one lexical token; the org name dominates the embedding) — that distinction is lexical and the
negative guards are load-bearing anti-fab logic an NN must not re-derive. The 18-skill-direct classifier
is an **arm the §7 bake-off must PROVE** (confusion matrix + no anti-fab regression) before adoption.

## 6. Anti-fabrication guardrails (G2)
- **No mixed numeric answers.** Exact answer types are answered PURELY by the deterministic skill with
  LLM-compose suppressed — preserve the shipped `is_deterministic` + `deterministic_suffix` skip-compose
  path VERBATIM (`message_handler` ~L317-336).
- **Resolver re-runs the negative guards** (§5 step 3) — a "KG/role" family signal must NOT dispatch
  "how do I become a dean"; the process/leaf-dept/bare-name guards are dispatch preconditions, with the
  process-vs-identity cases as committed regressions. [RAG SF-2 / router S-3]
- **Terminal-skill precondition** (§5 step 3) — the SF-1 fix above.
- **Empty result → RAG, never honest-partial** (except the two terminal skills, now gated).
- **Wrong-but-valid hazard** — thin margin on ambiguous entities → clarify; the ambiguous-entity set is
  **DB-derived** (surname/acronym collisions via `persons_by_lastname`/org candidates), not hand-seeded.
  [RAG SF-3]
- **Sanitized live queries** (public text only). **Abstention everywhere** (low score/margin → clarify/RAG).

## 7. DATA-FIRST: Phase 0 bake-off (decides the mechanism honestly — not priors)
**Labeled routing set** — a few hundred real/realistic student messages labeled with correct route
(family + skill/source) + required slots, sourced from analytics logs + `eval/questions.txt` + authored
adversarial paraphrases + ambiguous/underspecified cases + the resolver hard-negatives (§6).
**MANDATORY train/test split** [router S-1, the biggest methodology fix]: exemplars drawn from one fold,
accuracy measured on a **paraphrase-disjoint** held-out fold (cluster by embedding; split by cluster) so
the exemplar-NN is not evaluated on its own training data (which would rubber-stamp classify-first).
**Bake-off (offline, before committing the build):** three arms — (1) detector-first (today + cue fix),
(2) **coarse-classify → deterministic within-KG** (the default prior), (3) classify-INCLUDING-18-skills.
**Metrics:** family accuracy; KG **skill accuracy + per-skill confusion matrix** (reveals the
faculty/people/officer smear); structured-false-negative rate; margins; clarify precision.
**BLOCKING anti-fab gate** [RAG SF-4]: an arm that raises `false-honest-partial` OR `wrong-confident-exact`
above the detector-first baseline is REJECTED regardless of any accuracy win. Arm #3 is adopted only if
it beats arm #2 on skill accuracy WITH no anti-fab regression.

## 8. Latency model (router C-1 — was asserted, now a real build requirement)
Today the structured path pays **ZERO** encode (regex + SQL only). Classify-first adds a ~20-80ms nomic
**Ollama HTTP** encode to it. Requirements:
- **`query_vec` seam (BUILD TASK):** thread an optional `query_vec` through `V2RetrieverShim.retrieve` →
  `V2Retriever.retrieve` and skip its internal encode (currently ~L310) when supplied. Until this lands,
  the "share the encode with the retriever" claim is unbacked. (§15 checklist.)
- **ONE encode per message (hard invariant)** — the single vector serves the classifier and (on
  fall-through) the retriever. Unit test asserts `embed_query` ≤ 1× per `handle()`.
- **Structured/KG path latency answer** — the FAST-PATH (§5 step 1) is **first-class**, sized from the
  Phase-0 head distribution to keep common KG intents at zero-encode; queries beyond it accept a measured
  +20-80ms, with a **p95 budget + go/no-go gate** stated in §12. (Not "optional optimization.")
- **Encode-prefix/version guard** [RAG Con-2 / router S-4 / K4] — the router classifier needs the RAW
  message under a FIXED router prefix; the retriever uses `search_query:` + may rewrite, so a shared
  *differently-prefixed/rewritten* vector decalibrates thresholds → the shared vector must be the raw
  router-prefixed one, retriever re-encodes if its query differs. Route exemplars carry encoder **+ prefix
  + model** version; startup fails loudly on mismatch (incl. `EMBEDDING_MODEL`/`OLLAMA_URL` drift).

## 9. Eval & calibration (BOTH goals)
Per-route thresholds calibrated by sweep on the held-out fold. **Margin (top1−top2) is PRIMARY**; the
absolute cosine floor is secondary and per-route (nomic L2-normalized query vectors with the prefix put
unrelated short queries at 0.4–0.6, so a global 0.5 floor is fragile — seed only). **G2 metrics:**
`P(structured→RAG/chat/web)` ≈ 0 (top); `unsafe-RAG exact-answer rate` = 0; family accuracy; KG skill
accuracy + confusion matrix; slot exact-match. **G1 metrics:** within-RAG outcome accuracy (food/event/
general); RAG↔live **recall** (+ the BM25-off-on-filter recall impact, §4); **false-honest-partial rate**;
prose abstention/over-route rate; clarify precision; **command-layer precedence regression** on the
existing intent_detector cases [RAG Con-3]. Exemplars start ~15-25 positive/family + 8-12 hard negatives
on the KG-vs-prose boundary (seed: "dean's welcome message", "how to become an officer", "what's on the
GSA website", "how do I cite a paper") — **counts are calibrated for the coarse-family world and must be
re-sized if arm #3 wins** [router S-4]. Grow from shadow logs.

## 10. Phasing
- **Phase 0 — data + bake-off (§7).** Labeled set (with train/test split), three-arm bake-off, pick the
  mechanism. Deliverable: dataset + bake-off report. No production change.
- **Phase 1 — unified Router (no LLM on path).** Command layer + first-class fast-path + coarse classifier
  + deterministic resolver (with negative guards + terminal-skill + inverse-FN guards) + clarify + the 3
  RAG outcomes + RAG↔live + the `query_vec` seam. Flag-gated, shadow-mode first, then flip.
- **Phase 2 — LLM slot-recovery (sub-flag).** Only if shadow shows the KG-classified-but-unresolved
  residue is non-trivial. Additive; zero-regression when off.

## 11. Testing (TDD)
Unit tests per component (exemplar KNN, classifier accept/margin, resolver slot-validation +
candidate-IDs-only + empty→RAG, the negative-guard preconditions, terminal-skill guard, inverse-FN guard,
clarify templates, command precedence, one-encode invariant). Regression set runs THROUGH `handler.handle`
(NOT `scripts/ask.sh` — bypasses the gate; assert this). The negative-guard cases ("how do I become a
dean/officer", "who is responsible for officer elections", "departments in math", the meta-sentence
surname trap) and the known-failing live set ("top 10 by citations in mechanical engineering",
role/area paraphrases) become committed regressions that must stay green. Every change grows
`eval/questions.txt`.

## 12. Rollout (gated, reversible)
Feature flag `ROUTER_V21` (default OFF) + Phase-2 sub-flag. **Shadow mode first:** log new-vs-current on
real traffic + the labeled set; measure G1+G2 before flipping. **Flip gate is conjunctive** [router S-5]:
the new wins land AND structured-false-negative rate ≤ current AND the §11 regressions green AND no NEW
wrong-confident-exact/false-honest-partial AND p95 latency within budget. Flag stays as kill-switch. Code
change → `scripts/restart.sh`; route-store/DB changes → no restart.

## 13. Decisions (who said what)
- **Primary mechanism:** coarse-family classifier + deterministic within-KG selection (DEFAULT); 18-skill
  classifier only if the §7 bake-off proves it without anti-fab regression. [both experts' prior]
- **Unify the 4 routers → one module:** YES. **Intent-detector split:** commands deterministic up front,
  content to the classifier. **Within-RAG:** 3 outcomes via existing handlers/`source_type`; fine-grained
  deferred. **Roll-own** reaffirmed (reject semantic-router + fine-tuned head). **Veto:** SUBSUMED by
  classify-first (resolved). **Clarify:** first-class, deterministic. **LLM gate:** deferred Phase 2.
- **NEW from re-review:** terminal-skill precondition (SF-1); resolver re-runs negative guards (SF-2/S-3);
  §4 RAG taxonomy corrected to 3 outcomes + BM25 caveat (C-1); `query_vec` seam + one-encode + fast-path
  first-class + p95 gate (router C-1); bake-off train/test split + anti-fab blocking gate + confusion
  matrix (S-1/SF-4); inverse-FN guard (Con-5); margin-primary thresholds (S-4); DB-derived ambiguous set
  (SF-3); command-precedence + prefix/version guards (Con-2/3).

## 14. Non-goals (YAGNI)
NOT text-to-SQL / LLM-generated SQL. NOT a new heavy dependency (no semantic-router, no Haystack, no
second embedding model, no fine-tuned head in Phase 1). NOT GraphRAG auto-extraction. NOT an LLM call on
the hot path in Phase 1. NOT fine-grained per-corpus RAG sub-routing (DEFERRED — flagged §4/§15). NOT
preserving any current module for its own sake — but DO preserve `route()`'s negative guards + the
`is_deterministic` numeric path as load-bearing anti-fab logic.
